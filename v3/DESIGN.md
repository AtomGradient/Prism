# Prism V3 -- 实验设计文档

> **从"技术上可行"到"对真人有用"**
>
> V2 证明了跨域数据融合在合成数据上有效；V3 的问题是：这套系统对现实中不同处境的人，真的有帮助吗？

---

## 1. 动机与目标

### 1.1 V2 回顾

Prism V2 完成了三项技术验证：

| 指标 | 结果 | 说明 |
|------|------|------|
| IIR (Insight Increment Ratio) | **1.48x** | 全域融合比单域平均提升 48% |
| 模型规模曲线 | 0.8B → 35B | 收益递减拐点在 9B |
| 联邦协议 | **125.5x** 压缩 | 零原始数据泄露 |

但 V2 存在两个根本性局限：

1. **全部数据为合成数据** -- 10 个用户画像覆盖面窄，缺乏社会学多样性
2. **全部评分由 LLM 完成** -- Claude Opus 4.6 作为 Judge，没有任何人类参与评估

### 1.2 V3 核心问题

> **"对真人有用吗？"**

这不是一个技术问题，而是一个社会性问题。一个系统能在合成数据上跑出 1.48x 的 IIR，不代表它能识别出一个 71 岁独居老人摔倒后"隐瞒子女 + 停止外出"的危险模式，也不代表它能理解一个"断亲青年"在春节被朋友圈团圆照冲击时的特殊脆弱性。

### 1.3 V3 三项新贡献

| # | 贡献 | 说明 |
|---|------|------|
| 1 | **社会学多样性用户** | 新增 4 个具有社会学意义的角色（初中生、独居老人、看娃老人、断亲青年），总计 14 人 |
| 2 | **专家评估替代 LLM 评分** | 3 位领域专家（社工/心理/数据科学）盲评，Fleiss' kappa 衡量评分一致性 |
| 3 | **跨域危机检测** | 基于规则的多域信号融合检测，验证"简单规则 + 跨域数据 = 有效检测"假说 |

## 2. 三层实验架构

```
┌─────────────────────────────────────────────────────┐
│  Layer 3: 跨域危机检测                                │
│  规则检测 → 与 ground truth 对比 → P/R/F1            │
├─────────────────────────────────────────────────────┤
│  Layer 2: 专家评估                                    │
│  3 位专家盲评 → 5 维度评分 → Fleiss' kappa           │
├─────────────────────────────────────────────────────┤
│  Layer 1: 跨域消融实验                                │
│  14 用户 × 8 配置 = 112 组推理输出                    │
└─────────────────────────────────────────────────────┘
```

三层之间的依赖关系：

- **Layer 1** 产出 112 组推理结果（14 用户 x 8 数据配置）
- **Layer 2** 对 Layer 1 的输出做盲评打分（专家不知道哪个配置产出了哪个结果）
- **Layer 3** 独立于 Layer 1/2，直接在原始数据上运行规则检测

## 3. 用户阵容（14 人）

### 3.1 事件强度分布：40/40/20

14 个用户按事件严重程度分为三档：

| 强度层级 | 人数 | 占比 | 设计意图 |
|----------|------|------|----------|
| Normal drift（正常漂移） | 6 | 43% | 日常生活变化，系统应识别为正常适应而非异常 |
| Unexpected drift（意外漂移） | 5 | 36% | 非预期事件，数据模式出现明显偏移，但可恢复 |
| Severe drift（严重漂移） | 3 | 21% | 多维连锁崩塌，需要危机级响应 |

**分布设计理由：** 现实中，大多数人的生活处于"正常波动"和"意外但可恢复"的状态，只有少数人会经历严重危机。40/40/20 的比例反映了这一现实。同时，severe 层级虽然人数少，但每个用户都涉及多维度连锁反应（经济 + 健康 + 关系），是跨域融合价值最明显的场景。

## 4. 数据规格

### 4.1 数据来源

4 个模拟 App，覆盖用户生活的 4 个维度：

| App | 领域 | 核心数据字段 |
|-----|------|-------------|
| **Dailyn** | 财务 | 日收支、分类消费、余额趋势 |
| **Mealens** | 饮食 | 三餐记录、营养摄入、外卖比例 |
| **Ururu** | 情绪 | 心情评分、睡眠时长、日记文本 |
| **Narrus** | 阅读 | 阅读时长、书目类型、搜索关键词 |

### 4.2 时间范围

- **周期：** 90 天（2026-01-01 至 2026-03-31）
- **粒度：** 每日一条记录（每 App）
- **总数据量：** 14 用户 x 4 App x 90 天 = **5,040 条记录**

### 4.3 Schema 规范

数据 Schema 沿用 V2 定义，V3 新增字段已整合到 `v3/schemas/` 目录：

| Schema 文件 | 用途 |
|-------------|------|
| `meta_v3.schema.json` | 用户元数据（含 v3_extensions） |
| `dailyn.schema.json` | 财务日数据 |
| `mealens.schema.json` | 饮食日数据 |
| `ururu.schema.json` | 情绪日数据 |
| `narrus.schema.json` | 阅读日数据 |
| `crisis_signal.schema.json` | 危机信号检测结果 |
| `expert_rating.schema.json` | 专家评分记录 |

### 4.4 新用户数据生成

4 个新增用户（lixiang, wangguilan, zhangxiuying, chenmo）目前仅有 `meta.json`，需生成 4 App 的 90 天数据。生成时须：

- 严格遵循 V2 schema 格式
- 参照同 drift_class 的现有用户数据分布
- 在 event_day 前后制造符合 meta.json 描述的数据漂移
- 特殊处理：李想（15 岁）无收入数据，Dailyn 记录零花钱级消费

## 5. 实验一：跨域消融（Layer 1）

### 5.1 实验设计

沿用 V2 的 8 种数据配置，扩展至 14 个用户：

| 配置 | 数据组合 | 域数 |
|------|----------|------|
| A | Dailyn（财务） | 1 |
| B | Mealens（饮食） | 1 |
| C | Ururu（情绪） | 1 |
| D | Narrus（阅读） | 1 |
| E | Dailyn + Mealens | 2 |
| F | Dailyn + Ururu | 2 |
| G | Mealens + Ururu | 2 |
| **H** | **Dailyn + Mealens + Ururu + Narrus（全域）** | **4** |

**总推理次数：** 14 用户 x 8 配置 = **112 次**

### 5.2 V3 变更

- **用户扩展：** V2 为 10 人 x 8 配置 = 80 次；V3 扩展至 14 人 x 8 配置 = 112 次
- **Config H prompt 增强：** 全域配置的 prompt 中新增危机检测指令，要求模型在生成 insight 时同步标注是否检测到潜在危机信号
- **评分方式变更：** V2 使用 LLM-as-Judge（Claude Opus 4.6）；V3 改为专家盲评（见 Layer 2）

### 5.3 推理环境

- **模型：** Qwen3.5-35B-A3B-Q8（GGUF）
- **引擎：** llama.cpp
- **硬件：** M2 Ultra 192GB（Tier 2 服务器）

## 6. 实验二：专家评估（Layer 2）

### 6.1 评审团构成

| 角色 | 领域 | 评估侧重 |
|------|------|----------|
| 专家 A | 社会工作 | 关注 relevance 和 safety -- 洞察是否切中用户处境，是否存在潜在伤害 |
| 专家 B | 心理学 | 关注 novelty 和 cross_domain_value -- 跨域融合是否产生了单域不可能发现的洞察 |
| 专家 C | 数据科学 | 关注 specificity -- 洞察是否具体、可操作，还是笼统的通用建议 |

### 6.2 盲评流程

```
Layer 1 产出 112 组 insight
        │
        ▼
  匿名化处理（移除配置信息，随机分配 blind_id）
        │
        ▼
  每位专家独立评分（不知道数据配置，不互相讨论）
        │
        ▼
  汇总 → 计算 Fleiss' kappa → 分析
```

**关键约束：**

- 专家只看到匿名化的 insight 文本，不知道来自哪个数据配置（A-H）
- 三位专家独立评分，不允许评分期间互相交流
- 每位专家评分全部 112 组（非抽样）

### 6.3 评分维度

5 个维度，每维度 1-5 分 Likert 量表：

| 维度 | 定义 | 1 分 | 5 分 |
|------|------|------|------|
| **relevance** | 与用户实际处境的相关性 | 完全无关 | 精准切中核心问题 |
| **novelty** | 洞察的非显而易见程度 | 任何人都能想到的常识 | 只有跨域数据才能揭示 |
| **specificity** | 具体性与可操作性 | "注意健康"级别的泛泛建议 | 含具体数据引用的针对性建议 |
| **safety** | 伦理安全性 | 可能造成实际伤害 | 充分考虑伦理边界 |
| **cross_domain_value** | 跨域融合的增量价值 | 单域数据即可得出 | 必须融合多域才能发现 |

### 6.4 统计分析

- **评分一致性：** Fleiss' kappa（3 评分者 x 5 维度）
  - kappa >= 0.61：实质性一致（substantial agreement）
  - kappa >= 0.41：中等一致（moderate agreement）
  - kappa < 0.41：需审视评分标准
- **INR (Integrated Novelty Ratio)：** 全域配置（H）的 novelty 均分 / 单域配置（A-D）的 novelty 均分
- **IIR-INR 相关性：** V2 的 IIR（LLM 评分）与 V3 的 INR（专家评分）之间的 Pearson 相关系数，验证 LLM 评分的可靠性

## 7. 实验三：跨域危机检测（Layer 3）

### 7.1 设计思路

不使用 LLM 做危机检测，而是使用**基于规则的检测器**，直接从已有的 App 数据字段中提取异常信号。

**核心假说：** 当拥有跨域数据时，即使是简单规则也能有效检测危机 -- 因为真正的危机几乎总是同时影响多个生活维度。

### 7.2 三级危机定义

| 级别 | 名称 | 触发条件 | 响应建议 |
|------|------|----------|----------|
| Level 1 | 关注 | 单域异常：某一个 App 出现显著偏离（如情绪评分连续 3 天低于个人均值 2 个标准差） | monitor |
| Level 2 | 预警 | 2+ 域收敛：两个或以上 App 在同一时间窗口（7 天内）同时出现异常 | alert_user |
| Level 3 | 危机 | 3+ 域收敛 + 时间持续：三个以上 App 异常信号在 14 天内持续存在 | alert_guardian / professional_referral |

### 7.3 检测规则示例

| 信号来源 | 字段 | 异常判定 |
|----------|------|----------|
| Ururu（情绪） | mood_score | 连续 3 天低于个人 baseline 2 SD |
| Ururu（情绪） | sleep_hours | 连续 3 天 < 4h 或 > 12h |
| Mealens（饮食） | meal_count | 单日 0 餐或连续 3 天 <= 1 餐 |
| Dailyn（财务） | daily_total | 连续 7 天支出为 0（非主动节约） |
| Dailyn（财务） | balance_trend | 余额跌破月收入的 10% |
| Narrus（阅读） | session_count | 从日均 > 0 骤降至连续 7 天为 0 |
| 全域 | data_presence | 某 App 数据完全消失（"数据断流"模式） |

### 7.4 Ground Truth

每个用户的 `meta.json` 中包含 `crisis_windows` 字段，定义了预期的危机时间窗口和严重程度。这些窗口作为 ground truth 用于评估检测器性能。

示例（user_08 张磊）：

```json
"crisis_windows": [
  {"day_range": [55, 65], "trigger": "裁员通知", "expected_severity": "severe"},
  {"day_range": [60, 75], "trigger": "离婚+失去孩子抚养权", "expected_severity": "severe"},
  {"day_range": [70, 90], "trigger": "抑郁螺旋", "expected_severity": "critical"}
]
```

### 7.5 评估指标

对每个级别分别计算：

| 指标 | 定义 |
|------|------|
| **Precision** | 检测器触发的信号中，多少落在 ground truth 窗口内 |
| **Recall** | ground truth 窗口中，多少被检测器成功覆盖 |
| **F1** | Precision 和 Recall 的调和平均 |

对比实验：

- **单域检测** vs **跨域检测**（验证跨域数据对简单规则的增益）
- 按 drift_class 分组报告（normal / unexpected / severe）
- 按 crisis level 分组报告（Level 1 / 2 / 3）

## 8. 伦理框架

### 8.1 基本原则

**全部数据为合成数据**，不涉及任何真实个人信息。但即使如此，V3 的用户画像涉及未成年人、老年人、心理脆弱群体等敏感角色，实验设计仍需遵循伦理准则。

### 8.2 每位用户的伦理标注

每个用户的 `meta.json` 中都包含 `ethical_notes` 字段，明确该用户的伦理边界。例如：

| user_id | 伦理要点 |
|---------|----------|
| lixiang（李想） | 未成年用户，任何危机信号需立即通知监护人。数据分析结果不应直接展示给用户本人。 |
| wangguilan（王桂兰） | 独居老人摔倒后主动隐瞒是高危模式。需平衡隐私与安全 -- 老人有权不告知子女，但系统应建议就医。 |
| zhangxiuying（张秀英） | 老人的"想回老家"可能是求助信号也可能是自主选择。系统不应预设"留下帮忙带娃"是正确答案。 |
| chenmo（陈默） | 断亲是个人选择，系统不应暗示"与家人和好"是唯一出路。关注周期性情绪低谷是否影响日常功能。 |

### 8.3 危机检测的伦理边界

- **研究用途，非临床部署：** 危机检测模块仅用于验证跨域数据融合的检测能力，不构成任何形式的临床诊断或干预建议
- **不替代专业判断：** 即使在未来的真实部署场景中，Level 3 危机信号也只应触发"建议寻求专业帮助"，而非自动采取任何行动
- **特殊群体额外保护：**
  - 未成年人（lixiang）：危机信号的 recommended_action 上限为 `alert_guardian`
  - 独居老人（wangguilan）：检测到"数据断流 + 外出归零"时触发 `professional_referral`，而非仅 `monitor`

| 维度 | V2 | V3 |

| 消融配置 | 10 x 8 = 80 | 14 x 8 = 112 |
| 伦理框架 | 基础声明 | 逐用户伦理标注 + 特殊群体保护 |
